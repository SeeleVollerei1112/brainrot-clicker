# 运行、审查与修复指南

> 执行测试和审查修复阶段的完整参考。

---

## 一、运行流程

### 3 步线性流程

```
启动游戏 ──> 启动检查+注入测试 ──> AI轮询等待完成
```

### 步骤 1：启动游戏

```bash
# 清理旧日志
if exist log.txt del log.txt
# 启动游戏
.codemaker\editor-cli.exe --port 19836 exec "EditorAPI.run_game()"
# 等待 10 秒加载
ping -n 11 127.0.0.1 > nul
```

### 步骤 2：启动检查 + 注入测试

读 log.txt 检查启动错误（见下方启动检查表）。无错误则注入测试：

```bash
.codemaker\editor-cli.exe --port 19836 exec "require('test.test_{module_name}')" --platform framesync
```

### 步骤 3：AI 轮询等待完成

**严格执行固定节奏轮询，禁止根据预估耗时一次性等待。** 无论首次运行还是修复重跑，都必须遵守以下流程：

1. 注入后先等 **10 秒**（首次加载开销）
2. 每隔 **5 秒**执行一次检查命令
3. 直到检测到结束标记或超时

```bash
# 每轮检查（三条一起执行）
type log.txt | find "[TEST:END]"
type log.txt | find "stack traceback"
type log.txt | find "not found"
```

| 检测结果 | 含义 | 后续 |
|---------|------|------|
| 找到 `[TEST:END]` | 测试正常完成 | 截图 → 停止游戏 → 读日志 + 看截图 |
| 找到 `stack traceback` 或 `not found` | Lua 运行时崩溃 | 停止游戏 → 读日志定位 → 修复 → 重启 |
| 超时（delay 总和 + 30 秒） | 可能死循环或等待过长 | 停止游戏 → 检查逻辑 |

**禁止行为：**
- ❌ 因为"已知测试耗时"而跳过中间轮询、一次性等待到结束
- ❌ 修复重跑时用更长的等待间隔

**原因：** 固定 5 秒间隔能尽早发现崩溃（可能在第 1 秒就发生），避免无意义等待。

### 截图（测试完成后、停止游戏前）

测试正常完成后，在停止游戏之前截图：

```bash
.codemaker\editor-cli.exe --port 19836 exec "EditorAPI.take_screenshot()"
# 等待 2 秒确保截图保存
ping -n 3 127.0.0.1 > nul
```

然后停止游戏：

```bash
.codemaker\editor-cli.exe --port 19836 exec "EditorAPI.stop_game()"
```

> **先截图再停，先停后读日志。** 截图必须在游戏运行中执行，日志必须在停止后读取。

截图路径从 `log.txt` 中匹配含 `editor_screenshots` 的最后一行获取，读取图片进行视觉分析。若截图路径解析失败或图片不存在，标记 `[SCREENSHOT:SKIP]` 继续，不阻塞流程。

### 启动错误检查

| 匹配条件 | 处理 |
|---------|------|
| `attempt to` 或 `stack traceback:` | 停止 → 修复 → 重启 |
| `encoding is not valid` | 停止 → 修复非 UTF-8 → 重启 |
| `module '` + `not found` | 停止 → 修复 require 路径 → 重启 |

## 二、审查顺序

**日志存在错误时，必须先处理日志错误** 
**日志没有错误时，必须先审查截图，再分析断言日志。** 截图审查期间禁止引用断言结果（PASS/FAIL），禁止因"全部 PASS"而放松视觉检查。

```
正确流程: 日志无错误 → 截图 → 逐项视觉检查 → 记录截图结论 → 读日志断言 → 综合判断
错误流程: 读日志(看到37/0全PASS) → 带着"没问题"的心态看截图 → 走过场
```

---

## 三、日志分析

测试完成后读取全部日志。

### 日志标记

| 标记 | 含义 |
|------|------|
| `[TEST:BEGIN]` / `[TEST:END]` | 测试套件开始/结束 |
| `[TEST:PASS]` / `[TEST:FAIL]` / `[TEST:ERROR]` | 断言通过/失败/框架异常 |
| `attempt to` / `stack traceback` | Lua 运行时错误 |

### 错误来源

- 行号指向 `client/` → 修 UI 代码
- 行号指向 `test/` → 修测试代码
- 行号指向 `test_framework.lua` → 大概率传参问题，检查测试代码

### 截图路径

从 log.txt 匹配含 `editor_screenshots` 的最后一行，提取文件路径，直接读取图片进行视觉分析。

## 四、截图分析

### 截图异常零容忍

截图审查中发现**任何**视觉问题，必须标记为 ❌ 并进入修复流程。**禁止以"功能逻辑正确"或"断言 PASS"为由忽略视觉问题。** 如果审查备注中出现任何负面描述（如"对比度低"、"不太清晰"、"略有偏移"），该截图**必须判定为 FAIL**，不允许标 ✅ 后在备注里轻描淡写。

### 每轮独立诊断（禁止惯性沿用）

每一轮截图分析必须**独立于上一轮的结论**，禁止因"上一轮是位置问题，这一轮应该还是"而跳过重新分析。具体要求：

1. **禁止**在新一轮分析中直接复用上一轮的诊断结论
2. 必须**先看当前截图得出新结论**，再与上一轮对比确认问题是否变化
3. 如果上一轮的修复已部分生效（如元素已上移），必须重新判断**剩余问题的根因**是否与上一轮相同
4. **交叉验证**：当某个元素出现异常时，检查**相同坐标/相同属性的兄弟元素**是否也有同样问题。如果没有，说明根因不在共享属性（如位置）上，而在差异属性（如尺寸、文字长度）上

### 对抗性自问

每张截图审查前，**先强制问自己**：「如果这张图有问题，最可能是什么？」写出至少一个潜在风险点，然后带着这个怀疑去逐项检查。禁止跳过此步骤。

### 逐项强制检查清单

每张截图**必须逐项检查**以下内容，每项给出明确的 PASS / FAIL 判定，不得笼统带过：

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 1 | **文字可读性** | 文字与背景对比度是否足够？文字是否被遮挡？能否清晰辨认内容？白字白底、黑字黑底等均判 FAIL |
| 2 | **布局合理性** | 元素位置是否符合预期？居中的是否居中？对齐的是否对齐？ |
| 3 | **无异常重叠** | 元素之间是否有意外遮挡？z-order 是否正确？ |
| 4 | **无裁切异常** | 文字/图片是否被截断？内容是否溢出容器？**发现截断时必须区分原因**：①被**屏幕边界**截断（元素位于屏幕极边缘）②被**控件自身宽度/高度**截断（同一坐标的其他同类控件显示正常则指向此原因）③被**父节点区域**裁切。不同原因对应不同修复方向（调位置 vs 调尺寸 vs 调父节点），禁止笼统归因 |
| 5 | **颜色与样式** | 颜色是否符合预期？禁用态/激活态是否正确？透明度是否合理？ |
| 6 | **无渲染异常** | 是否有黑块、白块、闪烁、拉伸变形等渲染问题？ |
| 7 | **无不合理空白** | 是否有意外的大面积空白或元素缺失？ |

### 截图 + 断言综合判断

| 断言 | 截图 | 结论 | 处理 |
|------|------|------|------|
| PASS | 正常 | 通过 | — |
| PASS | 异常 | 需要修复 | UI 样式缺陷，检查修复 UI 代码 |
| FAIL | 正常 | 需要修复 | 检查修复 UI 代码 |
| FAIL | 异常 | 需要修复 | 检查修复 UI 代码 |

## 五、修复循环

最多 **5 轮**：分析（日志+截图）→ 修代码 → `stop_game()` → 重启+注入+运行。全部 PASS 且截图无异常 → 完成。

### 修复前必答清单（禁止跳过）

1. **效果链是否完整缺失？** 某操作的所有后续效果全部没发生 → **UI 代码事件回调逻辑**问题
2. **UI 代码逻辑是否正确？** 阅读完整回调逻辑，检查状态变量、条件判断、定时器参数
3. **只有确认 UI 逻辑无误后**，才可调整测试等待时间
4. **诊断不确定时先加日志验证，禁止直接改代码。** 对根因没有把握时，第一步是添加 `print` 输出关键变量的实际值，跑一轮收集证据，再决定修哪里
5. **修复前自检：每个改动是否与诊断一致？** 逐条检查即将提交的改动，回答"如果我的诊断是 X，这个改动是否从 X 逻辑推出？"。如果某个改动与诊断矛盾，必须删除该改动或修正诊断
6. **反事实验证：** 提交改动前，强制回答问题：「改动后**具体哪个视觉现象**会消失？」— 必须指向截图中观察到的具体问题，而非泛泛的"会好一些"。

### 修复方向

| 问题类型 | 修复方向 |
|---|---|
| 运行时错误（被测代码） | 修 UI 代码 |
| 运行时错误（测试代码） | 修测试代码 |
| 断言 FAIL — 回调效果完全缺失 | 修 UI 代码 |
| 断言 FAIL — 部分效果不符 | 修复 UI 逻辑 |
| 截图异常 | 调整 UI 布局、UI 样式等 |
| 超时 | 读日志定位 / 检查死循环 |

### 帧同步模式特殊修复规则

由于 ENode 无属性直读，修复时注意：

1. **节点不存在 (assertNotNil FAIL)**：
   - 检查 `GameAPI.create_eui_*` 的 name 参数拼写是否与测试中查找的名称一致
   - 检查节点创建是否在 `GAME_INIT` 事件回调中执行
   - 检查父节点是否存在（父节点 nil 会导致创建失败）

2. **层级关系错误 (assertChildExists FAIL)**：
   - 检查 `GameAPI.create_eui_*` 调用中 parent 参数是否指向正确的父节点
   - 注意变量名是否被后续赋值覆盖

3. **截图视觉异常（日志全 PASS 但截图异常）**：
   - 这是帧同步模式最重要的修复依据
   - 节点存在不等于显示正确——坐标、尺寸、颜色参数可能有误
   - 检查 `math.tofixed()` 是否正确应用（缺少 tofixed 会导致类型错误或位置偏移）
   - 检查 `Role.set_node_visible` 是否被正确调用（初始可能隐藏）

### 修改 UI 代码后同步 testspec

修复阶段**专注于快速修复和重跑**，不要求每轮修复后立即同步 testspec。待修复循环全部完成（所有测试通过 + 截图无异常）后，**统一**将修复期间的节点结构和交互逻辑变更同步到 `test/testspec_{模块名}.md` 及测试代码中的对应断言。

> 修复循环中只需确保测试代码与 UI 代码一致即可，testspec 文档的同步延后到循环结束。

### 第 4~5 轮强制换方向

前 3 轮一直改同一方向 → 必须切换（改 UI ↔ 改测试、改同文件 → 换文件、调等待 → 查逻辑）。

### Testspec 辅助修复

testspec 存在时，读"设计意图"和"交互逻辑"辅助判断：testspec 与代码一致但断言失败 → 测试问题；testspec 描述的行为未发生 → UI 代码 bug。

### 5 轮后仍有失败

输出最终报告：

```markdown
## 测试报告

| 项目 | 值 |
|------|-----|
| 模块 | {module_name} |
| 总轮次 | {N}/5 |
| 最终结果 | {PASS / PARTIAL / FAIL} |
| 通过/失败断言 | {passed} / {failed} |

### 未解决问题
- [ ] {FAIL name}: {description}

### 截图审查
| 截图 | 结果 | 备注 |
|------|------|------|

### 修复记录
| 轮次 | 修改文件 | 修改内容 | 效果 |
|------|---------|---------|------|
```
