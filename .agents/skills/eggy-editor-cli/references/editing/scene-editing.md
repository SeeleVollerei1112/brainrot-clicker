# 场景编辑

## 编辑器状态要求

**必须**处于编辑时才可以操作

## 思维准则

1. 明确目的
    - 叙事用途：这个场景想要传达什么情绪？（压抑、开阔、神秘、温馨）
    - 功能用途：玩家可以在场景内做什么？（战斗、解密、观景、探索）
    - 技术目标：需要使用多少组件？平均每个组件复杂程度（面数、定点数、渲染压力）是怎样的？
2. 收集参考
    - **如无特殊要求**，识别现有场景并分析，保证场景风格一致性
    - 真实场景 > 他人产出（避免抄袭，避免风格趋同）
3. 场景布局
    - 确认组件摆放的基准或者大概位置
    - 必要时可以使用方块组件先制作灰模
4. 场景组装
    - 大 → 中 → 小：地标 → 主要物体 → 细碎装饰
    - **如无特殊要求**，避免完美对称

## 空间坐标系统

统一使用**左手**坐标系以及**列向量**表示的坐标

- 将旋转四元数应用于某个向量：`Q * v`
- 将变换矩阵应用于某个向量：`M * v`

### 平移（ position ）与缩放 （ scale ）

| 分量 | 描述 | 正方向（第一视角）| 负方向（第一视角）|
|---|---|---|---|
| X | 水平横向 | 右 | 左 |
| Y | 垂直 | 上 | 下 |
| Z | 水平纵向 | 前 | 后 |

- EditorAPI 使用
  - 设置某个组件的平移：`%CLI% exec "print(EditorAPI.set_unit_attr(<unit_id>, {'position'\|'scale'}, math.Vector(<x>, <y>, <z>)))"`
  - 设置某个组件的平移：`%CLI% exec "print(EditorAPI.get_unit_attr(<unit_id>, {'position'\|'scale'}))"`

### 旋转 （ model_angle ）

| 分量 | 描述 | 正方向（第一视角）| 负方向（第一视角）|
|---|---|---|---|
| X | 俯仰角 | 低头 | 抬头 |
| Y | 偏航角 | 右转 | 左转 |
| Z | 滚转角 | 逆时针翻滚 | 顺时针翻滚 |

- 旋转方向统一规则
  - 正值：延轴向正方向逆时针旋转
  - 负值：延轴向正方向顺时针旋转
- 旋转分量应用的顺序为 Z → X → Y
- EditorAPI 使用（统一使用**弧度制**）
  - 设置某个组件的旋转：`%CLI% exec "print(EditorAPI.set_unit_attr(<unit_id>, 'model_angle', math.Vector(<x>, <y>, <z>)))"`
  - 获取某个组件的旋转：`%CLI% exec "print(EditorAPI.get_unit_attr(<unit_id>, 'model_angle'))"`

## 场景内组件数据获取

### 组件 ID 获取

- 按组件名称搜索：`%CLI% exec "print(EditorAPI.query_unit_ids(<name>, <use_regex>))"`
- 获取当前选中的组件的：`%CLI% exec "print(EditorAPI.get_selected_unit_ids())"`
- **不推荐**获取场景内所有组件的：`%CLI% exec "print(EditorAPI.get_all_unit_ids())"`

### 按组件 ID 获取组件数据

- 按属性名获取组件属性：`%CLI% exec "print(EditorAPI.get_unit_attr(<unit_id>, <attr_name>))"`
- 获取单一组件所有属性：`%CLI% exec "print(EditorAPI.get_scene_unit_data(<unit_id>))"`

## 组件预设信息查询流程

> ⚠️ **严禁编造 key 或依赖记忆中的 key 值**，必须通过文档查询。

### 查询步骤

1. 提取、拓展关键词
    - 用户说"树" → 搜索词："树木"或"植物"
    - 用户说"墙" → 搜索词："墙体"或"围墙"
    - 用户说"地板" → 搜索词："地板"或"地面"
2. 使用 `retrieve_knowledge` 技能根据关键词查询
3. 从结果中选择
    - 优先选：语义（参考视觉描述）吻合 + 尺寸适中 + 非机关类
    - 避免选：陷阱/弹射/传送等功能性机关（除非用户明确要求）
4. 提取必要的组件信息
    - 组件编号（prefab_key）
    - 包围盒半长
    - 包围盒中心点

### 回退策略

| 情形 | 处理方式 |
|------|---------|
| 查到多个候选 | 选语义最匹配的，在告知用户时注明所用组件名称 |
| 查到但尺寸不合适 | 用 scale 调整到合适尺寸 |
| 完全查不到 | 向用户说明，请用户提供具体 key，不得猜测 |

## 组件空间信息

- 在编辑器中存在两种“大小”的概念
  - 世界坐标（ position ）：组件在空间中的位置
  - 欧拉角（ model_angle ）：组件的旋转
  - 缩放（ scale ）：组件相对于基础大小的缩放值
  - 尺寸（ size ）：组件在空间中的绝对大小（缩放后）
  - AABB 中心点（ aabb_center ）：组件真实 AABB 的中心点
  - AABB 尺寸（ aabb_size ）：组件真实 AABB 大小

### 获取场景内组件的空间信息

`%CLI% exec "print(EditorAPI.get_unit_space_info(<unit_id>))"`

```lua
{
  'position': Vector3(),    -- 世界坐标
  'model_angle': Vecotr3(), -- 旋转（弧度制）
  'scale': Vector3(),       -- 缩放
  'size': Vector3(),        -- 实际大小（缩放后）
  'aabb_center': Vector3(), -- AABB 中心点（世界坐标）
  'aabb_size': Vector3(),   -- AABB 大小（缩放后）
}
```

### 通过预设信息近似计算

- 组件预设信息中的“包围盒半长”代表了该组件在没有任何旋转（`(0.0, 0.0, 0.0)`）、缩放（`(1.0, 1.0, 1.0)`）时 AABB 尺寸的一半
- 组件预设信息中的“包围盒中心点”代表了该组件位于原点（`(0.0, 0.0, 0.0)`）和没有任何旋转（`(0.0, 0.0, 0.0)`）、缩放（`(1.0, 1.0, 1.0)`）时 AABB 中心点的位置
- 由于没有办法获取组件的所有顶点数据，无法计算出准确的的 AABB 或 OBB，只可使用旋转后的原始 AABB 去近似计算

    > 旋转后的 AABB 的 X 轴：旋转 x [包围盒半长[1], 0.0, 0.0]
    > 旋转后的 AABB 的 Y 轴：旋转 x [0.0, 包围盒半长[2], 0.0]
    > 旋转后的 AABB 的 Z 轴：旋转 x [0.0, 0.0, 包围盒半长[3]]
    > 旋转后的 AABB 的 8 个顶点：中心点 +/- 旋转后的 AABB 的 X 轴 +/- 旋转后的 AABB 的 Y 轴 +/- 旋转后的 AABB 的 Z 轴

### 使用示例

**需求：**
场景内原点（`(0.0, 0.0, 0.0)`）有一块草坪，用户要求在草坪的中心放一棵树
草坪组件的空间信息如下：

```lua
{
  'position': Vector3(0.0, 0.0, 0.0),
  'model_angle': Vecotr3(0.0, 0.0, 0.0),
  'scale': Vector3(1.0, 1.0, 1.5),
  'size': Vector3(7.5, 1.0, 7.5),
  'aabb_center': Vector3(0.0, 0.0, 0.0),
  'aabb_size': Vector3(7.5, 1.0, 7.5),
}
```

要放置的树的预设信息如下：

  - 包围盒半长：=`(8.52, 9.82, 7.8)`
  - 包围盒中心点：`(-0.37, 9.82, 0.94)`

❌ **错误做法（直接用 y=0 放树）：**
草坪的顶面 `grassTopY = 0 + ((0 + 1) x 1) = 1`
设想将树放在 `(0.0, 0.0, 0.0)` 的位置同时不使用任何旋转（`(0.0, 0.0, 0.0)`）与缩放（`(1.0, 1.0, 1.0)`）
树的底面 `treeBottomY = 0.0 + (9.82 - 9.82) x 1.0 = 0.0`
`treeBottomY < grassTopY` → 树的底面将陷入草坪

✅ **正确做法：**
草坪的顶面 `grassTopY = 0 + ((0 + 1) x 1) = 1`
设想将树放在 `(0.0, 1.0, 0.0)` 的位置同时不使用任何旋转（`(0.0, 0.0, 0.0)`）与缩放（`(1.0, 1.0, 1.0)`）
树的底面 `treeBottomY = 1.0 + (9.82 - 9.82) x 1.0 = 1.0`
`treeBottomY == grassTopY` → 树的底面将正好落在草坪上
使用 EditorAPI 将树放置在设想的位置：`%CLI% exec "EditorAPI.create_obstacle(103488, math.Vector3(0.0, 1.0, 0.0))"`

## 组件对称性

组件不一定对称性，这意味着一个组件绕某一个轴旋转 180 度后的样子和没有旋转时的样子不同。在进行特定摆放的时候，需要考虑组件的对称性，如果没有组件对称性的确定性信息，请默认组件是没有对称性的。因此，在做对称摆放（例如镜像摆放）的时候，多个物体不能使用同样的旋转角度

### 使用示例

**需求：**
在以景内原点（`(0.0, 0.0, 0.0)`）为中心，半径为20的圆上均匀摆放 4 个椅子，并且使他们都朝向原点

要放置的椅子有以下属性：

  - 包围盒半长：`(1.5, 1.25, 2.0)`
  - 包围盒中心点：`(0.0, 1.25, 0.0)`

❌ **错误做法（直接用 y=0 放树）：**
在 X 和 Z 轴的正负方向上各放置一个椅子就可以满足需求
椅子在 Z 轴方向上最长，椅子的朝向一定是延 X 轴的，在 X 上放置时不需要旋转
在 X 轴上摆放 2 个椅子，分别在 `(20.0, 0.0, 0.0)` 和 `(-20.0, 0.0, 0.0)`，没有旋转
在 Z 轴上摆放 2 个椅子，分别在 `(0.0, 0.0, 20.0)` 和 `(0.0, 0.0, -20.0)`，绕 Y 轴旋转 90°
因为椅子关于 YZ 平面是不对称的，造成一定有一半的栅栏不是朝向原点的，一定不满足需求

✅ **正确做法：**
在 X 和 Z 轴的正负方向上各放置一个椅子就可以满足需求
椅子在 Z 轴方向上最长，椅子的朝向一定是延 X 轴的
在 X 轴上摆放 2 个椅子，分别在 `(20.0, 0.0, 0.0)` 和 `(-20.0, 0.0, 0.0)`，分别绕 Y 轴旋转 0° 和 180°
在 Z 轴上摆放 2 个椅子，分别在 `(0.0, 0.0, 20.0)` 和 `(0.0, 0.0, -20.0)`，分别绕 Y 轴旋转 90° 和 270°
因为椅子具体的朝向信息是未知的，4 个椅子一定会同时朝向原点或者同时背对原点，可能满足用户需求

## 组件创建

### 规则

- 创建组件相关 API 只支持创建时的位置，旋转缩放需要在创建后再设置
- 单次最多创建 **10 个组件**

  | 组件总数 | 处理方式 | 预计耗时 |
  |---|---|---|
  | ≤ 20 个 | 直接分批创建，无需特别提示 | ~4 秒 |
  | 21~50 个 | 分批创建，执行前告知用户预计等待时间 | ~10 秒 |
  | > 50 个 | 考虑合并为更大组件减少数量；若仍需大量创建，告知用户较长等待 | 较长 |

### 示例代码

```batch
# 第一批（最多10个组件）
%CLI% exec "EditorAPI.create_obstacle(<prefab_key_1_1>, math.Vector3(<x_1_1>,<y_1_1>,<z_1_1>))"
%CLI% exec "EditorAPI.create_obstacle(<prefab_key_1_2>, math.Vector3(<x_1_2>,<y_1_2>,<z_1_2>))"
# ... 最多10条

powershell -Command "Start-Sleep -Milliseconds 200"

# 第二批
%CLI% exec "EditorAPI.create_obstacle(<prefab_key_2_1>, math.Vector3(<x_2_1>,<y_2_1>,<z_2_1>))"
%CLI% exec "EditorAPI.create_obstacle(<prefab_key_2_2>, math.Vector3(<x_2_2>,<y_2_2>,<z_2_2>))"
# ...
```
