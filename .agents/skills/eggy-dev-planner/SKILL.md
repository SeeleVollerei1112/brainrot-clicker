# 游戏开发管理

## 核心原则

1. **范围优先**：先判断输入规模，再选择计划模板
2. **自适应粒度**：游戏级 / 系统级 / 功能级三档，自动匹配
3. **文件持久化**：计划保存为 Markdown 文件，支持持续更新
4. **增量更新**：更新计划时不覆盖已完成条目

---

## 范围评估与粒度选择

### 自动判断规则

| 判定粒度 | 输入特征 | 示例 |
|---------|---------|------|
| **游戏级** | 描述包含完整玩法循环、多个子系统、整体目标 | "做一个4人竞技生存游戏" |
| **系统级** | 描述聚焦于某一较大功能模块（含多个子功能） | "做一个完整的背包和商店系统" |
| **功能级** | 描述仅涉及单一、明确的功能点 | "做一个积分榜" / "加一个复活机制" |

### 手动覆盖指令

开发者可在描述中加入以下关键词强制指定粒度：

- `/游戏级` → 强制使用游戏级模板
- `/系统级` → 强制使用系统级模板
- `/功能级` → 强制使用功能级模板

## 计划模板

### 🎮 游戏级模板（Phase 里程碑）

适用于完整游戏开发，生成 3~5 个阶段。

```markdown
# 开发计划：{游戏名称}

> 创建时间：{日期} | 范围：游戏级 | 粒度：Phase 阶段

## 概述

{一句话描述游戏目标}

---

## Phase 1：{阶段名称}（核心玩法验证）

**目标**：{本阶段完成后可验证什么}

**主要任务**：
- [ ] {任务1}
- [ ] {任务2}
- [ ] {任务3}

**验收标准**：
- {可测试的验收条件1}
- {可测试的验收条件2}

---

## Phase 2：{阶段名称}（依赖 Phase 1 完成）

**目标**：{本阶段完成后可验证什么}

**主要任务**：
- [ ] {任务1}
- [ ] {任务2}

**验收标准**：
- {可测试的验收条件}

---

<!-- 重复至 Phase N（建议3~5个） -->

## 📊 进度总览

| Phase | 名称 | 状态 | 依赖 |
|-------|------|------|------|
| 1 | {名称} | ⬜ 未开始 | 无 |
| 2 | {名称} | ⬜ 未开始 | Phase 1 |
```

---

### 🔧 系统级模板（模块拆解）

适用于单个较大系统，拆解为若干模块。

```markdown
# 开发计划：{系统名称}

> 创建时间：{日期} | 范围：系统级 | 粒度：模块拆解

## 系统概述

{系统的核心功能和边界描述}

---

## 模块 1：{模块名称}

**功能**：{模块的职责描述}
**依赖**：{依赖的其他模块，无则填"无"}

**子任务**：
- [ ] {子任务1}
- [ ] {子任务2}

---

## 模块 2：{模块名称}

**功能**：{模块的职责描述}
**依赖**：模块 1

**子任务**：
- [ ] {子任务1}
- [ ] {子任务2}

---

## 📊 模块依赖图

{模块1} → {模块2} → {模块3}
```

### ✅ 功能级模板（TODO 条目）

适用于单一功能点，生成扁平 TODO 列表。

```markdown
# TODO：{功能名称}

> 创建时间：{日期} | 范围：功能级 | 粒度：TODO 条目

## 任务列表

| 优先级 | 任务 | 状态 | 验收标准 |
|--------|------|------|---------|
| 🔴 高 | {任务1} | ⬜ | WHEN {条件} THEN {期望结果} |
| 🟡 中 | {任务2} | ⬜ | WHEN {条件} THEN {期望结果} |
| 🟢 低 | {任务3} | ⬜ | WHEN {条件} THEN {期望结果} |

## 详细 TODO

- [ ] **[高]** {任务1描述}
  - **验收**：WHEN {条件}，THEN {期望结果}

- [ ] **[中]** {任务2描述}
  - **验收**：WHEN {条件}，THEN {期望结果}

- [ ] **[低]** {任务3描述}
  - **验收**：WHEN {条件}，THEN {期望结果}
```

## 文件保存规则

### 命名约定

```
plan-<kebab-case-name>.md
```

**命名示例**：

| 输入描述 | 文件名 |
|---------|--------|
| "无尽跑酷游戏" | `plan-endless-runner.md` |
| "背包商店系统" | `plan-inventory-shop.md` |
| "积分榜功能" | `plan-leaderboard.md` |

- 使用游戏/系统/功能的英文 kebab-case
- 若开发者使用中文名，AI 自动转换为合适的英文 kebab-case
- 若开发者指定文件名，优先使用开发者指定的名称

### 保存位置

保存至**项目根目录**（与 `memory.md` 并列）。

完成后告知开发者：`计划已保存至 plan-xxx.md`

## 计划更新操作

### 标记任务完成

将指定任务的 `- [ ]` 改为 `- [x]`，其余条目不变。

```markdown
- [x] **[高]** 实现积分计算逻辑   ← 已完成
- [ ] **[中]** 实现积分榜 UI       ← 未完成
```

### 追加新任务

在适当位置追加新条目，**已完成的条目 `[x]` 保持不变**：

```markdown
- [x] 已完成的任务        ← 不动
- [ ] 原有未完成任务
- [ ] 新追加的任务        ← 追加在合适位置
```

### 更新阶段状态

更新进度总览表中的状态标记：

| 状态标记 | 含义 |
|---------|------|
| ⬜ 未开始 | 尚未启动 |
| 🔄 进行中 | 已开始未完成 |
| ✅ 已完成 | 全部子任务完成 |

---

## 执行流程

1. **接收描述** → 判断/确认粒度
2. **选择模板** → 填充内容
3. **保存文件** → 告知路径
4. **后续更新** → 增量修改，不覆盖已完成项

---

## 注意事项

1. 若输入描述附带 `eggy-game-design` 生成的设计文档，直接读取"系统拆分建议"章节作为任务来源
2. 游戏级计划中，Phase 1 应始终包含**最核心的可验证玩法**（MVP）
3. 所有任务描述须具体可操作（避免"完善系统"等模糊表述）
4. 验收标准须可测试，使用 WHEN/THEN 格式
